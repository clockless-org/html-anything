# html-anything

[![skills.sh](https://skills.sh/b/clockless-org/html-anything)](https://skills.sh/clockless-org/html-anything)

> An agent skill that turns any file or URL into an **interactive HTML
> infographic** — the AI analyzes the content, finds the patterns, and
> designs a single self-contained page that shows you *what's actually
> there*. Not a format converter.

```
WhatsApp export → relationship timeline + topic clusters + reply-time graph + emoji map
CSV (50K rows)  → summary charts + sortable table + outlier callouts
Long PDF        → key claims extracted + section nav + a 5-min summary view
GitHub repo     → architecture diagram + file tree + the 3 files that explain it
```

Drop in a file, get back a `.html` you can open, email, share, or host.

## Install

Pick whichever:

```bash
npx skills add clockless-org/html-anything                # via the open-skills CLI
```

```bash
git clone --depth 1 https://github.com/clockless-org/html-anything ~/.claude/skills/html-anything
```

Either works. Claude Code picks up [`SKILL.md`](./SKILL.md) automatically.
Other agents (Codex, Cursor, Cline, …) that follow the
[Agent Skills](https://agentskills.io) standard work the same way.

Then in any session:

> Convert this WhatsApp export to HTML: ~/Downloads/_chat.txt

> Render https://github.com/clockless-org/html-anything as a one-page explainer

> Make a webpage out of this CSV: data.csv

## Standalone CLI mode (without Claude Code)

For non-Claude-Code workflows, the same pipeline runs as a Node CLI.
Currently clone-to-run; npm publish coming.

```bash
git clone https://github.com/clockless-org/html-anything
cd html-anything
npm install
export ANTHROPIC_API_KEY=sk-ant-...   # or OPENAI_API_KEY=sk-...
npx tsx src/cli.ts ~/Downloads/_chat.txt
```

The CLI uses the same [`prompts/`](./prompts) as the skill mode — they
share a single source of truth.

## Supported sources

Each source has a dedicated prompt in
[`prompts/`](./prompts) describing layout decisions and the
data shape Claude should design for.

| Prompt | Triggered by | Example output |
|---|---|---|
| [`whatsapp.md`](./prompts/whatsapp.md) | `_chat.txt` exports (detected by content) | bubble timeline, sender filter, date jump |
| [`csv.md`](./prompts/csv.md) | `.csv`, `.tsv` | sortable table, virtualized for big data, summary charts |
| [`markdown.md`](./prompts/markdown.md) | `.md`, `.markdown` | reading view + TOC sized to the doc |
| [`github-repo.md`](./prompts/github-repo.md) | `github.com/owner/repo` URLs | repo explainer, file tree, README, key files |
| [`url-article.md`](./prompts/url-article.md) | blog posts, articles, news pages | stripped reading view, pulled quotes |
| [`json.md`](./prompts/json.md) | `.json` | structured viewer chosen by shape (array → table, object → key-value, tree → collapsible) |
| [`default.md`](./prompts/default.md) | catch-all | content-shape-based fallback |

## How it works

```
┌──────────┐    ┌──────────┐    ┌──────────┐
│ file/URL │───▶│ identify │───▶│  design  │───▶  output.html
│          │    │  source  │    │ via LLM  │
└──────────┘    └──────────┘    └──────────┘
```

1. **Identify the source.** File extension, URL pattern, or content sniff.
2. **Load the matching prompt.** `prompts/<source>.md` describes how
   to design for this kind of content.
3. **Sample, don't slurp.** Read ~5–15 KB worth of representative content.
   The full data goes inline in the output, not into the LLM prompt.
4. **LLM designs the page.** It writes the HTML+CSS+JS based on the
   sample, schema, and source-specific guidance. Output is a single
   self-contained file.
5. **Inline the data.** A `__DATA__` placeholder in the LLM's HTML gets
   substituted with the full data as a JSON literal. The page renders
   the full thing client-side from the JS the LLM wrote.

This split — LLM designs the experience, code injects the data — is what
makes the same skill work on a 50-row CSV and a 500K-row CSV without
hitting the LLM's context limits.

## Contributing

The fastest contribution is **a new source prompt**. Most of the
interesting work happens in markdown, not code.

### Adding a source (prompt-only, the common case)

Pick a source whose current viewers make you sad — a `.eml` archive, a
Slack export, a custom log format, anything.

1. Add `prompts/<your-source>.md`. Use any existing prompt as a
   template — they're all ~30–60 lines:
   - **Layout decisions by shape** — what does the LLM pick for small vs
     large vs single-sender vs multi-sender, etc.?
   - **Always include** — search, copy, dark mode, responsive layout.
   - **Data shape** — what fields the LLM can reference in `DATA`.
   - **Tone** — typography and visual register that fits the content.
2. (Optional) Add `src/parse/<your-source>.ts` if the CLI mode needs a
   custom parser. Skill mode reads files via Claude Code's tools, no
   parser needed — only the prompt.
3. Open a PR. Include a small synthetic input (anonymous / fake data
   only) under `examples/<your-source>/input.<ext>` so we can regenerate
   the demo.

That's the whole loop. No framework changes for new sources.

### Adding a parser (CLI mode, less common)

If a format needs special parsing logic before sampling (PDF text
extraction, binary archives, big files needing streaming), add a
`Parser` in `src/parse/`. See [`docs/CONTRIBUTING.md`](./docs/CONTRIBUTING.md)
for the contract. Most formats won't need this.

### Improving an existing prompt

The prompts are markdown. Iterate on layout decisions, fix typos, add
edge-case handling. PRs welcome. Run `npm run examples` to regenerate
the demo outputs and confirm the change reads well.

### What we won't merge

- Format converters that produce a fixed template regardless of content
  (that's pandoc — already exists).
- Parsers that cap output at a small subset (the full data has to be
  inlined; the page should not need to call back to a service).
- Anything that breaks the "single self-contained .html" rule.
