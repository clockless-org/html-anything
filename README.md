# html-anything

> Turn any file into the **best possible HTML reading experience for that
> specific content** — designed by an LLM, rendered as a single
> self-contained file.

```
$ html-anything chat.zip
→ designing page…
→ chat.html  (1 file, opens in any browser)
```

`html-anything` is **not** a format converter. Pandoc and a thousand other
tools already do that — and they produce *correct* but *plain* output.

This tool's premise is different: **the same content type deserves
different presentations depending on its shape**. A 2-person friend chat
shouldn't render the same as a 200-person Slack channel. A 5-page recipe
PDF shouldn't render the same as a 500-page report. The right reading
experience for *this specific file* is a design decision, not a
template lookup.

So the LLM does the design. It looks at a sample of the input and the
schema, decides what UI helps you read this thing, and writes the
HTML+CSS+JS. The full data is inlined into that HTML — the LLM only ever
sees a sample, never the whole file. The output is a single
self-contained `.html` you can email, share, or host anywhere static
files live.

Inspired by [Thariq's "Unreasonable Effectiveness of HTML"][thariq] and
the [HKUDS/CLI-Anything][cli-anything] thesis that **tomorrow's users
are agents**: if HTML is what humans read best, agents should produce
HTML — *for this content*, not for this format.

[thariq]: https://x.com/trq212
[cli-anything]: https://github.com/HKUDS/CLI-Anything

## How it works

Two steps. Cleanly separated.

```
┌─────────────┐    ┌──────────────┐    ┌──────────────┐
│  your file  │───▶│   parser     │───▶│  htmlize     │───▶  output.html
│             │    │  (per-format │    │  (LLM-driven │
│             │    │  → JSON)     │    │  page design)│
└─────────────┘    └──────────────┘    └──────────────┘
```

1. **Parse**: a per-format parser turns the file into a `ParsedFile`:
   `contentType`, a `summary`, a representative `sample` (a few records),
   the full `data`, and `meta` stats.
   - Cheap, deterministic, no LLM. ~50-100 lines per format.
2. **htmlize**: the LLM gets `contentType` + `summary` + schema + `sample`
   and produces a complete HTML document with a `__DATA__` placeholder.
   We substitute the placeholder with the full data inlined as JSON.
   - The LLM never sees the full file. The full data is rendered
     client-side from the inlined JSON by the JS the LLM wrote.

This split is the whole point:
- The LLM designs *the experience for this file*.
- The code injects *the data*.
- The output is **single-file, self-contained, reproducible** once
  generated, and works on a 50-row CSV the same way it works on a
  500K-row CSV (the LLM only sees the sample either way).

## Install

```bash
npm install -g html-anything
# or
npx html-anything <file>
```

Set one of:
```bash
export ANTHROPIC_API_KEY=sk-ant-...
# or
export OPENAI_API_KEY=sk-...
```

## Usage

```bash
html-anything chat.zip                       # → chat.html
html-anything paper.pdf --out paper.html
html-anything data.csv --title "Q1 sales"
html-anything story.md --model claude-opus-4-7
```

The LLM call costs roughly $0.01–$0.05 per file with Claude Sonnet 4.6
(default), depending on size of the sample. The **output** is reusable
forever — no LLM calls happen when someone opens the HTML.

## Built-in parsers

| Parser | Extensions | What the LLM sees |
|---|---|---|
| `markdown` | .md, .markdown | headings + opening 1500 chars + counts |
| `whatsapp` | .txt (detected) | first 8 + last 4 messages + sender stats |
| `csv` | .csv, .tsv | header + sample rows + column stats |
| `json` | .json | shape description + first/last entries |
| `text` | .txt, .log, others | opening + closing + line count |

More parsers coming. **Or write your own** —
see [`docs/CONTRIBUTING.md`](./docs/CONTRIBUTING.md). The whole
architecture is plugin-shaped.

## Contributing

The fastest contribution: a new parser for a format that produces
ugly output today. Pick one that bugs you (Slack export, mbox archive,
PDF, Gmail Takeout, your custom log format) and write a ~50-line parser.

See [`docs/CONTRIBUTING.md`](./docs/CONTRIBUTING.md) for the parser
contract. Everything else (rendering, layout decisions, interactivity)
is the LLM's job.
